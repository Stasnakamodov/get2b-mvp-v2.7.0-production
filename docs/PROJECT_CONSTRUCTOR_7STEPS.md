# GET2B: Конструктор проектов (7 шагов)

> Актуальная сборка: `v2.7.0-production` | Дата: 2026-04-11

## Обзор

Конструктор проектов — основной flow платформы GET2B. Клиент создаёт проект в 7 шагов, менеджер управляет процессом через Telegram-бота. Два режима входа:

- **Ручной** — клиент заполняет всё с нуля
- **Из каталога** — товары и поставщик подтягиваются из корзины каталога

---

## Архитектура

```

  ├── CreateProjectContext.tsx     — React-контекст, общее состояние 7 шагов
  ├── CartLoader                  — заполнение из корзины каталога
  ├── ProjectIdLoader             — восстановление из БД (перезагрузка/возврат)
  ├── TemplateLoader              — загрузка из шаблона
  └── Step1..Step7                — компоненты шагов

hooks/
  ├── useProjectSupabase.ts       — CRUD проектов (create, save, load, updateStep)
  ├── useProjectSpecification.ts  — работа с project_specifications
  ├── useTelegramNotifications.ts — форматирование и отправка уведомлений

lib/
  ├── types/project-status.ts     — статусы и допустимые переходы

  ├── telegram/                   — TelegramService + ManagerBotService

```

## Шаги

### Шаг 1: Карточка компании

| | |
|---|---|
| **Файл** | `Step1CompanyForm.tsx` |
| **Статус** | `draft` |
| **Что делает клиент** | Заполняет реквизиты компании: название, ИНН, КПП, ОГРН, юр. адрес, банковские реквизиты, контакты |
| **Что сохраняется** | Создаётся запись `projects` со статусом `draft`. Данные компании — в `projects.company_data` (JSONB) |
| **Telegram** | Уведомление менеджеру с данными компании |

**Особенности:**
- Автосохранение с debounce 2 сек при изменении полей
- Можно загрузить карточку компании (PDF/фото) — OCR извлекает данные
- Валидация 13 обязательных полей перед переходом
- Можно загрузить данные из профиля или шаблона

---

### Шаг 2: Спецификация / Инвойс

| | |
|---|---|
| **Файл** | `Step2SpecificationForm.tsx` |
| **Статус** | `draft` → `in_progress` |
| **Что делает клиент** | Создаёт спецификацию (вручную или из каталога) или загружает файл инвойса |
| **Что сохраняется** | Строки в `project_specifications`, файлы в storage `step2-invoices` |
| **Telegram** | Форматированная спецификация + кнопки одобрения/отклонения менеджеру |

**Особенности:**
- Два режима: ручное создание позиций (`create`) или загрузка файла (`upload`)
- Каждая позиция: наименование, код, кол-во, ед. изм., цена, итого, фото
- При flow из каталога — позиции заполнены автоматически
- Поллинг статуса каждые 4 сек (ожидание одобрения менеджера)
- При отклонении (`receipt_rejected`) — клиент может исправить и отправить заново

---

### Шаг 3: Загрузка чека об оплате агенту

| | |
|---|---|
| **Файл** | `Step3PaymentForm.tsx` |
| **Статус** | `in_progress` → `waiting_approval` → `waiting_receipt` / `receipt_rejected` |
| **Что делает клиент** | Загружает чек/квитанцию об оплате агенту |
| **Что сохраняется** | Файл в storage `step3-supplier-receipts`, URL в `projects.receipts` |
| **Telegram** | Чек + кнопки одобрения/отклонения менеджеру |

**Особенности:**
- После загрузки — ожидание одобрения менеджером через Telegram
- При одобрении → автопереход на шаг 4
- При отклонении → toast-уведомление, повторная загрузка

---

### Шаг 4: Выбор способа оплаты

| | |
|---|---|
| **Файл** | `Step4PaymentMethodForm.tsx` |
| **Статус** | `receipt_approved` → `filling_requisites` |
| **Что делает клиент** | Выбирает один из 3 способов оплаты |
| **Что сохраняется** | `projects.payment_method` |
| **Telegram** | Уведомление о выбранном способе оплаты |

**Способы оплаты:**

| Метод | ID | Комиссия | Время |
|---|---|---|---|
| Банковский перевод | `bank-transfer` | от 0,5% | 1-3 дня |
| P2P перевод | `p2p` | 1% | до 1 часа |
| Криптовалюта | `crypto` | 0% | до 30 мин |

**Система рекомендаций:**
1. Если `supplierData` содержит `payment_methods` (из каталога) — карточки подсвечиваются зелёным "Реквизиты поставщика"
2. Если нет — **echo-механизм**: поиск по имени поставщика в завершённых проектах + в `catalog_verified_suppliers`
3. При единственном доступном методе — автовыбор

---

### Шаг 5: Реквизиты для оплаты

| | |
|---|---|
| **Файл** | `Step5RequisiteSelectForm.tsx` |
| **Статус** | `filling_requisites` → `waiting_manager_receipt` |
| **Что делает клиент** | Выбирает существующие реквизиты или вводит новые |
| **Что сохраняется** | `project_requisites` (JSONB), опционально — новая запись в `bank_accounts` / `supplier_cards` / `crypto_wallets` |
| **Telegram** | Форматированные реквизиты менеджеру |

**Маппинг методов → таблиц:**

| Метод оплаты | Таблица реквизитов | Тип |
|---|---|---|
| `bank-transfer` | `bank_accounts` | `bank` |
| `p2p` | `supplier_cards` | `p2p` |
| `crypto` | `crypto_wallets` | `crypto` |

**Особенности:**
- Предзаполнение из `supplierData` (данные каталога)
- При одном реквизите поставщика — автовыбор
- Выбор страны: Китай / Турция / Другая (влияет на поля формы)
- Сохранение в адресную книгу для повторного использования

---

### Шаг 6: Ожидание чека от менеджера

| | |
|---|---|
| **Файл** | `Step6ReceiptForClient.tsx` |
| **Статус** | `waiting_manager_receipt` → `in_work` |
| **Что видит клиент** | Лоадер → "Чек получен и готов" с ссылкой на скачивание |
| **Что сохраняется** | Менеджер загружает файл → `step6-client-receipts` storage, URL в `projects.receipts` |
| **Telegram** | Автоматический запрос менеджеру на загрузку чека |

**Flow со стороны менеджера:**
1. Менеджер получает сообщение в Telegram с кнопкой "📤 Загрузить чек для клиента"
2. Нажимает кнопку → бот отправляет сообщение-запрос
3. Менеджер отправляет файл (фото/документ) в reply на это сообщение
4. Webhook: скачивает файл → загружает в storage → обновляет `projects.receipts` → меняет статус на `in_work`

**Защита от дублирования:**
- `localStorage` + `sessionStorage` + `useRef` — запрос менеджеру отправляется только один раз

**Поллинг:**
- Каждые 5 секунд проверяет наличие `manager_receipt` в `projects.receipts`
- При обнаружении → автоматически меняет статус на `in_work` → переход на шаг 7

---

### Шаг 7: Подтверждение клиента

| | |
|---|---|
| **Файл** | `Step7ClientConfirmationForm.tsx` |
| **Статус** | `in_work` → `waiting_client_confirmation` → `completed` |
| **Что делает клиент** | Просматривает чек от менеджера, загружает свой чек о получении средств, подтверждает завершение |
| **Что сохраняется** | Файл в `step7-client-confirmations`, URL в `projects.client_confirmation_url` |
| **Telegram** | Файл чека клиента менеджеру с подписью "КЛИЕНТ ЗАГРУЗИЛ ЧЕК!" |

**Особенности:**
- Чек клиента опционален — можно завершить проект без загрузки
- Файлы отправляются в Telegram через multipart upload (прямая загрузка binary)

## Flow: Каталог → Корзина → Проект

```
1. Клиент выбирает товары в /dashboard/catalog
2. Товары сохраняются в sessionStorage.project_cart
3. Клиент нажимает "Создать проект"
4. Редирект → /dashboard/create-project?from_cart=true
5. CartLoader:
   ├── Читает sessionStorage.project_cart
   ├── Маппит товары в формат спецификации
   ├── Загружает данные поставщика: loadCatalogSupplier(supplier_id)
   ├── setSupplierData() → контекст (для шагов 4-5)
   └── setMaxStepReached(2) → разблокирует шаг 2
6. Шаг 1: клиент заполняет данные компании
7. Шаг 2: спецификация уже заполнена из корзины
8. Шаги 3-7: стандартный flow
```

**Данные поставщика из каталога** (`catalog_verified_suppliers`):

```
supplierData = {
  name, company_name,
  payment_methods: ["bank_transfer", "p2p", "crypto"],
  bank_accounts: [{bankName, accountNumber, swift, cnapsCode, transferCurrency}],
  crypto_wallets: [{address, network, currency}],
  p2p_cards: [{card_number, holder_name, bank}],
  room_type: "verified" | "user"
}
```

## Telegram Webhook

**Endpoint:** `POST /api/telegram/webhook`

Обрабатывает:
- `callback_query` с `approve_*` / `reject_*` / `upload_supplier_receipt_*`
- Загрузку файлов (reply на сообщение с текстом "Загрузка чека для проекта" или "Проект:")
- Все файлы загружаются через multipart upload в Telegram (не через URL)
